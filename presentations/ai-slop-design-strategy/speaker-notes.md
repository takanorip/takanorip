# 登壇者ノート — AIスロップを避けるデザイン戦略

想定時間：**35〜45分**（質疑別）  
想定聴衆：デザイナー、デザインエンジニア、PM、エンジニアリングマネージャー

---

## 全体のメッセージ

今回の主張は、AIスロップを「見た目の流行り廃り」ではなく、**哲学がないままAIに判断を委ねることで起きる構造問題**として扱うこと。

キーフレーズ：

> AIは、哲学のない場所に平均を流し込む。

---

## オープニング（3分）

**スライド：タイトル → 今日の主張 → 構成**

### 話すポイント

- 「AIっぽいUI」は、紫グラデーションや角丸カードの問題に見えるが、本質はそこではない
- 今日の定義：**AIスロップなデザイン = 哲学がないデザイン**
- AIを否定する話ではない。むしろ、人間が判断を持つことでAIをより良く使う話

### 導入の問い

> そのデザインは、なぜその形であるべきかを説明できるか？

---

## 1. AIスロップなデザイン =「哲学」がないデザイン（6分）

**スライド：定義 → 症状 → デザイン哲学 → 哲学がないデザイン**

### 話すポイント

- 「紫グラデ」「3カード」などは症状。症状だけ潰しても別のスロップへ移動する
- デザイン哲学は大げさな理念ではなく、**迷ったときに戻る判断基準**
- 哲学がないと、レビューは好みの衝突になり、AIの初期出力が実質的な判断になる

### 補足例

- 「シンプル」は哲学ではない。「何を削ってもよく、何を削ってはいけないか」まで言えて初めて判断基準になる
- 「信頼感」は哲学ではない。「誰の、どんな不安を、どの表現で減らすか」が必要

---

## 2. 原因① — デザイン哲学を言語化できていない（7分）

**スライド：らしさは入力にならない → 曖昧語 → 判断の指定 → 5つの問い**

### 話すポイント

- チーム内では伝わる「らしさ」も、AIには暗黙知として伝わらない
- AIは空白を平均で埋める。だから「らしくして」は危険
- AIに渡すべきなのは、見た目の指定だけではなく**判断の指定**

### 強調する問い

1. ユーザーのどんな不安を減らすのか
2. どんな状態で意思決定してほしいのか
3. ブランドとして何を約束しないのか
4. 効率より守る体験は何か
5. 「これは違う」の境界線はどこか

### 使える一言

> 禁止リストは過去のスロップに効く。哲学は次のスロップにも効く。

---

## 3. 原因② — AIに任せる作業領域が間違っている（7分）

**スライド：4領域 → 任せると危険 → 任せるべき → 悪い分担/良い分担**

### 話すポイント

- AIを使うことが問題ではなく、**Whyの空白**をAIに埋めさせることが問題
- Why / What / How / Polish に分けると、AIに任せるべき領域が見えやすい
- AIは判断者ではなく増幅器。判断済みのものを速く広げる用途に向いている

### 悪い分担の説明

- 「いい感じに作って」→ AIが初期判断を握る
- 人間は最後に修正しているように見えるが、根本の方向性はAIが決めている

### 良い分担の説明

- 人間が哲学・制約・評価基準を決める
- AIが複数案を展開する
- 人間が選び、AIが整え、人間が責任を持つ

---

## 4. 人間が哲学を持ちAIが効率化するデザインプロセス（10分）

**スライド：全体像 → Philosophy → Principles → Brief → Exploration → Selection → Systematize → Production → Audit**

### 話すポイント

- 「作る前」に厚みを持たせ、「作った後」に検証する
- AI活用で速くするのは、探索・展開・検証。価値判断そのものではない
- 出力を選ぶ理由を残すことで、チームの学習資産になる

### 各ステップの説明

#### Philosophy
- AIに決めさせない領域
- 壁打ちはよいが、採用する判断は人間が持つ

#### Principles
- レビューで使える粒度に落とす
- 例：「判断を急がせない」「数字を盛らない」「不安を煽らない」

#### Brief
- AIに渡す前の設計書
- Goal / Audience / Principles / Avoid / Success Criteria を明確にする

#### Exploration
- 1案ではなく複数案
- 各案に「何を反映し、何を捨て、どんなリスクがあるか」を説明させる

#### Selection
- 好みではなく基準で選ぶ
- 選択理由を残す

#### Systematize
- 選んだ方向をトークン・ルール・プロンプト断片に変換する

#### Production / Audit
- AIで量産し、哲学・原則・アクセシビリティで検証する

---

## 5. デザインハーネスという幻想（6分）

**スライド：幻想 → ハーネスができること/できないこと → 哲学なき一貫性 → 正しい位置づけ**

### 話すポイント

- ハーネスとは、ルール・テンプレ・プロンプト・Lint・チェックリストのような安全柵
- ハーネスは必要。しかし、哲学の代替にはならない
- 「ルールは通っているが意思がないデザイン」は、AIスロップの別バージョン

### 伝えたい対比

| ハーネス | 哲学 |
|---------|------|
| 逸脱を減らす | 何を良いとするかを決める |
| 再現性を上げる | 例外を判断する |
| 最低限を守る | 責任を引き受ける |

### 使える一言

> ハーネスは哲学を運用する道具であって、哲学そのものではない。

---

## 6. カンムでの取り組み（7分）

**スライド：取り組み①〜⑥ → 目指す状態**

### 話すポイント

- カンムでは、AIを「デザインを置き換えるもの」ではなく、判断を補助し制作を速くするものとして扱う
- まず暗黙知を、AIにも人間にも読める形にする
- AIを使うほど、人間の責任範囲を明確にする

### 取り組みの柱

1. **デザイン哲学の言語化**  
   プロダクトとして大切にする判断、与えたくない不安、使わない表現を言語化する

2. **AIに任せる領域の分離**  
   ラフ案・文章候補・実装たたき台はAI、方向性・トーン・品質責任は人間

3. **AI用ブリーフ**  
   Context / User State / Principle / Do & Don't / Review Questions を先に書く

4. **レビューの問いを揃える**  
   好みではなく、「不安を減らしているか」「急かしていないか」などの問いで見る

5. **デザインシステムの再定義**  
   部品集から、AIにも渡せる判断基準へ

6. **小さく試し、レビューで学ぶ**  
   AI活用を、判断言語の学習ループとして扱う

### 差し替え候補

実際の社内施策やプロジェクト名があれば、以下を差し替えると具体性が増す。

- AI用ブリーフの実例
- レビュー会で使っている問い
- Design System / Figma / Cursor rules との接続
- AIで効率化できた作業例
- 失敗して学んだプロンプト例

## クロージング（2分）

**スライド：まとめ → 最後に → ありがとうございました**

### 話すポイント

- AIスロップは見た目の問題ではなく、哲学なき生成の問題
- AIは平均を出す。だから人間は哲学を言語化する
- 速さと品質を両立するには、AIに任せる範囲と人間が持つ責任を分ける

### 締めの一言

> AIは、哲学のない場所に平均を流し込む。  
> だからこそ、デザイナーは哲学を言語化する。
